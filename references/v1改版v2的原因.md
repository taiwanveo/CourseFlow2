**Cursor**

比對 CourseFlow 的視覺／動畫流程與 Web Video Presentation Skill 的差異。


你的觀察很準：**同一套 [Web Video Presentation Skill](https://github.com/ConardLi/garden-skills/blob/main/skills/web-video-presentation/SKILL.md) 在 CourseFlow 裡，和同事用 Claude Code 直接照 Skill 開發，產物差很多**。主因不是「Skill 失效」，而是 **兩條工作流根本不同**。

---

## 一句話差異

| | **同事：Claude Code + WVP Skill** | **CourseFlow 預設流程** |
|---|---|---|
| 產物 | 每章一個 **自寫 React 章節**（CSS / SVG / Canvas 動畫） | 每步一張 **可匯出 MP4 的投影片式畫面**（文字框 + 入場動畫 + 可選靜態配圖） |
| Agent 在做什麼 | 反覆讀 **CHAPTER-CRAFT**、按 step **逐步揭示**、做視覺演示 | 產 **大綱 / 口播稿**，視覺多半是 **預設版面 + 手動 Konva 編輯** |
| 動畫語意 | 「這一步數字長出來、那一步節點亮起」 | 「整步一起 `fade-up` / `scale-in`」 |

CourseFlow **有內建** WVP Skill 副本（`skills/`、`wvp-bridge`），但主要用在 **主題 token、入場／轉場目錄**，**不是**把 Skill 的 Phase 2「逐章寫 React 動畫」接進三階段流水線。

---

## 1. Skill 要求的是「章節級程式動畫」，CourseFlow 做的是「步驟級投影片」

[WVP Skill](https://github.com/ConardLi/garden-skills/blob/main/skills/web-video-presentation/SKILL.md) 的核心（見你 repo 內 `CHAPTER-CRAFT.md`）包括：

- **每章必須有 CSS / SVG / Canvas / JS 視覺演示**，禁止整章純文字  
- **清單要 1 項 = 1 step**，禁止一個 step 把 A/B/C 全部 stagger 上來  
- 用 **全局 step 計數器** 點擊推進，每步演不同戲  

同事用 Claude Code 時，Agent 會為 **每一章** 寫 `Chapter.tsx` + `narrations.ts`，在程式裡實作「橫條生長、節點點亮、對照切開」等。

CourseFlow 的 **一步（composition step）** 在預設是這樣建立的：

```72:94:packages/db/src/composition-sync.ts
export function defaultVisualForStep(stepId: string, screenContent: string): StepVisual {
  return {
    stepId,
    background: { type: "color", color: "#1a1a2e", opacity: 1 },
    elements: [
      {
        id: `${stepId}-hero`,
        type: "text",
        ...
        content: screenContent,
        fontSizePx: 64,
        ...
      },
    ],
    enterAnimationId: "fade-up",
    transitionId: "crossfade",
  };
}
```

也就是：**一大塊螢幕文字 + 一種入場動畫**。播放器（`CourseFlowPlayer`）也只有幾種入場 class（`fade-up`、`scale-in` 等），**沒有**「同一步內再點 5 次才揭示 5 個重點」的機制。

大綱 LLM 還明寫 **「不規劃動畫」**（`packages/llm/src/prompts.ts` 第 15 行），和 Skill 裡「outline 不寫動畫、章節開發時再設計」是同一哲學，但 CourseFlow **沒有後面的「章節開發」階段** 去補上。

---

## 2. 視覺 AI 很弱，且和 Skill 標準不一致

CourseFlow 的 `generateVisualDraft` 提示詞大致是：

```90:91:packages/llm/src/index.ts
export const VISUAL_SYSTEM_PROMPT = `你是教學簡報視覺設計助手。依 step 的 screenContent 與 infoPool，輸出 1920x1080 舞台的元素佈局 JSON。
不要改變文字內容。輸出 elements 陣列含 text 方塊位置與建議 enterAnimationId。`;
```

這是在排 **文字方塊位置**，不是 Skill 要求的 **「動起來的圖／演示」**。

另有 **AI 生圖**（`generate-step-image`）：單張 16:9 **插圖**，規則是 flat design、**不要文字**——適合背景或配圖，**不是**流程動畫、計數器、終端機模擬等 CHAPTER-CRAFT 範例。

`generate-visuals` 還依賴 **Worker 佇列**；本機常沒開 Worker 時，視覺階段更容易一直停在預設版面。

---

## 3. 匯出路徑：HyperFrames MP4 ≠ WVP 互動網頁

CourseFlow 終點是 **HyperFrames 編譯 → MP4**（`hf-bridge/compile.ts`）：每個教學步驟變成一個時間軸場景 + 入場動畫 + 字幕，本質是 **自動化的簡報影片**。

同事 Skill 的終點是 **Vite + React 演示站**（`localhost:5173`），用 **點擊 / `?auto=1` + 口播** 錄屏；動畫是 **React 狀態機 + CSS**，不是「每步一張靜態合成畫面」。

同一套美學標準（反 AI 味、逐步揭示）在 **可點擊網頁** 上好實作，在 **現有 composition 模型** 上要整段重做才接得上。

---

## 4. 同事流程裡還有「人機協作」，CourseFlow 偏自動化

Skill 的硬節點包括：

- **Checkpoint Plan**：對齊稿子、outline、主題、素材、開發模式  
- **第 1 章必須驗收** 當風格錨點  
- **CHAPTER-CRAFT 完工自檢**（不像 PPT、有視覺衝擊、禁止一次全展示）  

同事用 Claude Code 時，Agent 會被這些 checklist **逼著改到過關**。

CourseFlow 是 **文稿 → 語音 → 視覺 → 匯出** 產品流：快、可協作、可上 Render，但 **沒有**「每章寫完給人驗收再改」的內建節奏，預設產物就偏 **整齊但陽春** 的投影片。

---

## 5. 為什麼「OpenRouter / Edge 都一樣陽春」和 TTS 無關

TTS 只影響 **語音階段**。**陽春感幾乎全在視覺階段**：預設 hero 文字 + 主題色 + 簡單入場，除非你在 Konva 裡大量手調，或另走 WVP 腳手架自己寫章節。

---

## 你可以怎麼理解／取捨

**不是 Skill 比 CourseFlow 強，而是：**

- **Skill 路線** = 用 Agent **寫程式做教學動畫網頁**（品質高、慢、要會驅動 Agent）  
- **CourseFlow 路線** = 用平台 **結構化課程資料 + TTS + 主題 + MP4**（產品化、快，視覺模型偏簡報）

若要和同事接近，目前較接近的做法是：

1. 在 CourseFlow 把 **文稿拆更細**（一步一個可「點一下才講」的 beat），減少一屏塞滿條列。  
2. 視覺階段 **手動編輯** 或上傳素材，不要只靠預設 hero。  
3. 用 repo 的 **`pnpm sync-wvp` + WVP scaffold** 另開 `presentation/` 專案，用 Claude Code 照 Skill 做章節，再考慮是否錄屏當成片（這條 **尚未** 與 CourseFlow composition 自動合併）。  

若希望 **在 CourseFlow 內** 就達到 CHAPTER-CRAFT 水準，需要產品／架構級改動（例如：每章多 micro-step、程式化 visual block、或匯入 WVP 章節），不是只換 LLM 或主題就能解決。

---

若你願意切到 **Agent 模式**，可以一起規劃「最小可行」改法（例如：大綱強制拆 step、視覺提示詞對齊 CHAPTER-CRAFT、或 WVP 章節匯入路線）要動哪些模組。

## 為什麼值得保留 v1、另開 v2

**v1.0 繼續當「能用的產品」**

- 三階段（文稿 → 語音 → 視覺 → 匯出）已跑通，Render、TTS、登入、佇列等你也踩過一輪坑。
- 適合：**要快出課、要協作、要穩定 MP4**，視覺可以接受「主題 + Konva + 簡單入場」。
- 在 v2 大改架構時，v1 仍是**對外可 demo、可教學、可修 bug** 的基線，心理壓力也小。

**v2.0 適合當「重新對齊 WVP 方法論」**

- 若目標是接近 [Web Video Presentation Skill](https://github.com/ConardLi/garden-skills/blob/main/skills/web-video-presentation/SKILL.md) 那種**逐步揭示、章節自寫動畫、不像 PPT**，多半不是小修，而是：
  - 一步 ≠ 一張靜態 slide 的資料模型可能要變；
  - 播放器／匯出（HyperFrames MP4 vs 可點擊網頁錄屏）要選主路徑；
  - 視覺階段要接 **章節程式碼** 或至少 **micro-step**，而不是只靠 `defaultVisualForStep`。
- 在**複製專案**裡試錯，不會把 v1 的部署與使用者預期一起拖垮。

所以：**保留 v1 + 複製做 v2** 是常見且健康的做法，不是浪費。

## 複製時建議怎麼做（策略，不是要你現在就動手）

| 做法 | 優點 | 注意 |
|------|------|------|
| **新 Git repo**（例如 `CourseFlow-v2`） | v1 / v2 邊界最清楚，CI、Render 服務可完全分開 | 共用套件（`@courseflow/tts` 等）要決定 copy 還是抽成 npm / submodule |
| **同 repo 長期分支**（`main` = v1，`v2` 分支） | 歷史連續、可 cherry-pick 修復 | 久了容易混；Render 要兩套服務或只 deploy 一條線 |
| **Monorepo 裡 `apps/web-v2`** | 共用 packages 最省事 | 目錄變大，要嚴格規定「v1 不動、只改 v2」 |

以你「**明確要留住 v1.0**」來說，我會偏：

1. **新 repo 複製一份**（或 fork 成 private `CourseFlow-v2`），README 開頭寫清楚「v2 實驗、v1 在 xxx」；  
2. v1 的 `main` **只收 bugfix / 部署**，新功能大改一律進 v2；  
3. v2 第一個里程碑先**寫清楚產品取捨**（見下），再寫 code。

## v2 建議先定 3 件事（避免又做成「陽春 v1.5」）

3. **v1 要不要遷資料**

你現在看 `CHAPTER-CRAFT.md` 是對的方向：**v2 的靈魂應該是這份指引變成產品能力**，而不是再多幾個 `fade-up`。

## 風險與現實話

- **兩套程式碼** = 修 TTS / Supabase / Render 可能要修兩次，除非 v2 很快 rebase 或共用 packages。  
- **命名與心理**：資料夾叫 `CourseFlow-v2`、對外網域分開（例如 `courseflow-v2.onrender.com`），避免「以為還是同一套產品」。  
- v2 **不必**從頭重做 auth / DB；可複製 monorepo，先砍掉或擱置 v1 才需要的 Worker 複雜度，專心做視覺與節奏。
