## 為什麼要分開？

CourseFlow 內容頁的 AI 會把貼上的全文當成**教學原文**處理：

- **Markdown 匯入**（`MARKDOWN_TO_COURSE`）：要求 `script`（口播）以原文為底稿口語化，**不是摘要**；原文幾句，口播就應涵蓋幾句的資訊量。
- **大綱生成**（`OUTLINE`）：同樣從原文抽 step，並產出 `screenContent`（螢幕標語）。

因此若把「預期產出：圓餅圖」「測試 rise-soft」「`/api/health`」等**製作備註**寫進同一篇：

1. 可能被 AI 當成口播念出來（學員會聽到「預期產出圓餅圖」這類話）。  
2. 可能被寫進 `screenContent`（畫面出現驗收 jargon（術語））。  
3. 章節密度與切步會被「操作手冊」段落干擾。

**結論：學員文稿 = 只寫學員該聽的；製作指引 = 你自己對照，不進 AI。**

## 我能在教學文章裡指定「這段要什麼圖表／解說動畫」嗎？

### 今天（已上線）— 多為「內容觸發」，不是「明確指令」

| 方式 | 說明 | 可靠度 |
|------|------|--------|
| **口播裡帶數據** | 如 `40%`、`方案A：成本 12…` → 打包時 `inferVisualConfigFromText` 啟發式產 chart／table | 高（數字格式要對） |
| **章節關鍵字** | 標題或正文含「數據視覺」「方案對比」「完成率」等 → `isDataVisualChapter` 走 visual-mix | 中高 |
| **流程／清單用語** | 「先…然後…」「第一、第二…」→ 推斷 `flow`／`list-reveal` | 中 |
| **解說動畫** | 在 **Craft 配圖工作室**手動對某步產生；可與同步 chart 並存 | 手動，100% 可控 |
| **relationHint** | 大綱 API 的 step 欄位可帶 `list-reveal`、`contrast` 等（描述內容關係，非 CSS） | 目前主要由 AI 產出，**文稿裡寫了不保證會被解析** |

**今天沒有**在 Markdown 裡寫 `@chart: pie` 或 `<!-- 這步要解說動畫 -->` 就保證生效的語法；若把這類字寫進學員文稿，反而可能被念出來。

### 實務建議（現在就能用）

1. **要圖表**：把數字寫進**口播**（第四章學員文稿已示範），不要寫「請用 pie」。  
2. **要解說動畫**：文稿只寫概念；在 **Craft** 對該步按「產生解說動畫」（第五章見製作指引）。  
3. **要 Hook 三圖**：Checkpoint 上傳素材或工作室步驟 2–4 配圖，不靠文稿指定 URL。  
4. **要驗收某功能**：看製作指引對照表，不要寫進學員文稿。

### 以後可以怎麼做（尚未實作，可排產品）

若你希望「一篇 Markdown 裡偶爾夾製作備註」，較安全的設計是：

```markdown
## 第四章 數據視覺

:::cf-producer
step: 方案怎麼選 → table, sortBy=成本
step: 費用怎麼分 → chart:pie
:::

（以下才是學員會聽到的正文…）
```

或 HTML 註解 `<!-- cf:visual pie -->`，在 `generate-content` **解析前剝除**，只把指令寫進 `stepVisualConfigs`／Craft 計畫，**不進口播**。

這需要新增「製作指令區塊」解析器；你有興趣的話可以開 issue，我們再定語法與優先順序（啟發式 vs 明確覆寫）。

## 快速開始

1. 開啟 [`CourseFlow全功能驗收-學員文稿.md`](./CourseFlow全功能驗收-學員文稿.md)，複製 `## 開場` 到 `## 結語`。

3. 另開 [`CourseFlow全功能驗收-製作指引.md`](./CourseFlow全功能驗收-製作指引.md) 逐項驗收。
